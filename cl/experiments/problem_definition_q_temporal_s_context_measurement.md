# Problem definition: Q temporal grammar + S context/static fingerprint + subject measurement calibration

## One-line definition

This competition is not a clean “predict sleep quality from daily lifelog features” problem. A better working definition is:

> Predict a small set of self-reported binary sleep/life labels generated by a mixture of **Q-family temporal label grammar**, **S-family static/context fingerprint**, **subject-specific measurement calibration**, and **partial sensor-failure / observation-process anomalies**.

In other words, the observed labels are not direct measurements of one latent sleep-health state. They are the output of several overlapping processes:

```text
observed label
= true day state
+ subject-specific response style
+ previous/neighbor label grammar
+ static routine/context fingerprint
+ raw place/device/app context
+ sensor availability / wearing / logging artifacts
+ sparse label noise and transition-day exceptions
```

## Evidence summary

### 1. Q-family has real temporal grammar

Targets Q2/Q3 retain useful previous-label structure after subject control.

From `advanced76_temporal_influence_network.csv`:

```text
Q3: ALL_LAGS AUC 0.691, improvement vs subject-only +0.0409
Q2: ALL_LAGS AUC 0.683, improvement vs subject-only +0.0279
Q1: ALL_LAGS AUC 0.654, improvement vs subject-only +0.0028
```

Important temporal edges:

```text
Q3_lag -> Q3   +0.485
Q2_lag -> Q2   +0.467
Q1_lag -> Q3   +0.454
Q1_lag -> Q2   +0.314
```

Interpretation:

- Q2/Q3 are not just subject priors.
- They behave like temporally completed self-report items.
- Q1 appears to act as a precursor or grammar cue for Q2/Q3.
- Modeling Q2/Q3 independently loses structure.

### 2. S-family is more static/context/fingerprint-like

For S2/S3, strong apparent predictability often survives label shuffling or feature shifting within subject.

From `observation_negative_control_coverage_results.csv`:

```text
S3 coverage_plus_subject AUC 0.710
S3 subject_label_shuffle AUC 0.716
S3 feature_shift_plus_subject AUC 0.701

S2 coverage_plus_subject AUC 0.672
S2 subject_label_shuffle AUC 0.678
S2 feature_shift_plus_subject AUC 0.671
```

Interpretation:

- S2/S3 are heavily explained by subject/static routine/context.
- A model can look good on S2/S3 without learning day-level behavior.
- Temporal smoothing on S2/S3 is risky because it may repackage subject prevalence.

S4 is different from S2/S3: it shows context-event behavior, especially WiFi/BLE appearance.

From `advanced76_context_token_event_study.csv`:

```text
wifi_bssid:50:46:ae:5f:2e:14 appearance -> S4 rate 0.900 vs global 0.560
wifi_bssid:80:ca:4b:59:3b:52 appearance -> S4 rate 0.886 vs global 0.560
wifi_bssid:9e:25:4a:80:6f:61 appearance -> S4 rate 0.848 vs global 0.560
```

So the S-family should not be treated as one homogeneous hierarchy.

### 3. Subject-specific measurement calibration is large

A low-rank Bernoulli / IRT-style measurement model found that labels split into opposing axes rather than one severity direction.

From `advanced76_measurement_label_loadings.csv`:

```text
Q2  +7.39
Q3  +3.14
Q1  +0.83
S1  +0.12
S4  -2.07
S2  -3.17
S3  -5.83
```

Interpretation:

- Q-family and S-family are not simply positive indicators of one common sleep-quality state.
- Q2/Q3 and S2/S3/S4 may encode different response dimensions.
- Subject-level response style matters.

Subject residual means show that some subjects are much harder to explain with a global label grammar:

```text
High residual / irregular label language:
id05 3.62
id08 3.15
id10 2.94

Lower residual / more globally consistent:
id02 1.67
id06 1.86
```

Interpretation:

- A global model treats the same label as having the same meaning for everyone.
- That assumption is false or at least weak.
- Subject-specific calibration is not optional; it is part of the label-generation process.

### 4. Sensor-failure / observation-process anomalies are present but not the whole story

Coverage alone is not strong enough to explain the labels, but high-residual label rows are partly associated with coverage gaps.

From `advanced76_residual_anomaly_coverage_association.csv`:

```text
w_light_longest_missing_hours  AUC 0.622
hr_night_hours                 AUC 0.609
usage_longest_missing_hours    AUC 0.589
pedo_record_n                  AUC 0.583
```

Interpretation:

- Weird label rows are somewhat more likely on low-observability days.
- Some anomalies may be true transition days; others may be wearable/logging failure days.
- Sensor missingness should be modeled as a reliability/gating signal, not only imputed away.

## What makes the problem difficult?

### Difficulty 1. Labels are not one clean latent construct

If all labels measured one sleep-health state, then a single latent score or multi-output classifier would be natural. But the measurement loadings show opposing axes:

```text
Q2/Q3 positive axis
S2/S3/S4 negative axis
S1 near-neutral / mixed
```

This means the problem is partly a **measurement problem**, not just a prediction problem. The model must infer which label family is being generated by which mechanism.

### Difficulty 2. Target families have different causal/observational mechanisms

The targets should not be solved with one universal recipe:

```text
Q2/Q3: temporal grammar / self-report continuity
Q1: weak temporal precursor / app-context relation
S2/S3: subject/static routine/context fingerprint
S4: raw context event / WiFi-BLE-place signal
S1: mixed and less clearly defined
```

A single global feature model averages these mechanisms together and can improve CV while hurting the actual target mix.

### Difficulty 3. Subject identity is both useful and dangerous

Same-subject data makes subject priors valuable. But subject priors can also create fake confidence:

- If a target is mostly subject prevalence, CV can look good while day-level features are weak.
- If public/private hidden rows differ in state composition, subject-prior anchoring can fail.
- If context tokens are subject-specific places/devices, they can be strong but brittle.

This is especially problematic for S2/S3.

### Difficulty 4. Context identity tokens are high-signal but high-variance

Raw WiFi/BLE/app tokens contain real information. Some survive within-subject permutation, so they are not pure subject IDs. But they remain risky:

- sparse token counts,
- multiple testing,
- subject-specific places,
- unknown public/private distribution,
- possible leakage-like static context rather than general behavior.

Therefore they are useful as capped calibration or event features, not as unlimited model drivers.

### Difficulty 5. The observation process is entangled with behavior

In lifelog data, missingness is not random:

```text
HR missing = not wearing watch? charging? sleep? logging failure?
WiFi/GPS present = phone available? location? collection policy?
Screen/usage missing = asleep? no phone? OS background restriction?
```

So feature values and feature availability are both signals. But interpreting them is hard because sensor availability is partly behavioral and partly technical.

### Difficulty 6. Tiny data makes every sophisticated model unstable

There are only 450 labeled rows across 10 subjects and 7 binary targets. This creates several constraints:

- per-subject label estimates have high variance,
- token associations are sparse,
- multi-target models overfit easily,
- validation folds disagree,
- third-decimal leaderboard differences are not meaningful,
- BIC/regularization often disagree with apparent structural fit.

The dataset is large in raw logs but small in labels. Most raw signal cannot be safely supervised.

### Difficulty 7. Public-like validation is hard to define

The test rows are same-subject but not a simple future forecasting split. They are interleaved/nearby holes with both previous and next train rows for many cases. This means:

- using future/neighbor labels may be structurally legitimate under the competition split,
- tail-only validation can be misleading,
- random CV can overstate static-context features,
- public/private may reward different mixtures of interpolation vs context.

Validation is therefore part of the modeling problem.

### Difficulty 8. Anomalies are semantically ambiguous

High-residual rows can mean different things:

```text
true behavioral transition
self-report inconsistency
sensor missingness / device not worn
calendar alignment error
rare context event
label family contradiction
```

Without ground truth, anomaly handling must be probabilistic: lower confidence, gate feature families, or blend toward robust anchors.

## Working decomposition

A useful mental model is:

```text
For each subject s and day t:

Q_t = f_Q(previous Q labels, Q1/Q2/Q3 grammar, app/context, weak current sensors, subject calibration)

S2/S3_t = f_S_static(subject calibration, routine/context fingerprint, broad state, weak day signal)

S4_t = f_S4(context event, WiFi/BLE/app appearance, subject calibration, some temporal continuity)

observed_y_t = calibrated Bernoulli draw with anomaly/noise gate depending on sensor coverage and label residual risk
```

## Modeling implications

### Q2/Q3

Use:

- previous/neighbor Q labels,
- Q1 -> Q2/Q3 temporal edges,
- subject-calibrated temporal interpolation,
- app/context features only as small corrections.

Avoid:

- treating Q3 as pure noisy target to freeze completely,
- independent target modeling without Q-family grammar.

### S2/S3

Use:

- subject calibration,
- robust static context / routine summaries,
- conservative blending.

Avoid:

- aggressive temporal smoothing,
- over-trusting raw token features,
- interpreting high CV as day-level behavioral prediction.

### S4

Use:

- WiFi/BLE/app context event features,
- appearance/continuation/disappearance indicators,
- subject-debiased token filters,
- capped moves from stable anchor predictions.

Avoid:

- lumping S4 fully with S2/S3,
- letting sparse BSSID tokens dominate uncapped.

### Sensor-failure anomaly gate

Use coverage features to modulate confidence:

- long missing blocks,
- HR night missing,
- low pedometer/light coverage,
- unusual observation-process days.

Possible action:

```text
if anomaly risk high:
    reduce reliance on fine-grained context/temporal grammar
    blend toward subject-calibrated/base prior
else:
    allow target-specific mechanism features
```

## Research questions to solve next

1. **Can we separate subject calibration from true daily state?**
   - Need hierarchical model: subject intercepts + target-family slopes + day latent state.

2. **Are S2/S3 mostly response style or actual stable lifestyle state?**
   - Need compare subject-only, static-context-only, and day-varying-context models under subject-debiased validation.

3. **What exactly is S4 measuring?**
   - Context-event evidence suggests place/device state. Need inspect S4-positive token episodes and whether they align with sleep/usage behavior.

4. **Which anomalies should be trusted as transitions vs downweighted as noise?**
   - Need residual rows classified by coverage, context event, and neighbor label pattern.

5. **What validation scheme best approximates the leaderboard split?**
   - Need rank/interleaved split diagnostics and target-wise movement stability, not generic CV.

## Current best problem statement for future work

> This is a tiny-label, same-subject, multi-binary self-report reconstruction problem. Q labels are best treated as a temporal grammar / interpolation system. S2/S3 are dominated by subject/static context fingerprint and should be regularized heavily. S4 is a context-event target driven partly by raw WiFi/BLE/app identity. Subjects have different response calibrations, and high-residual days are partly tied to sensor coverage failures. The hard part is not feature extraction alone; it is disentangling day-level behavior from subject response style, static context, temporal label completion, and observation-process artifacts under a small and public/private-uncertain split.
